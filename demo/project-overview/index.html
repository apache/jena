Architecture

How It Works

SHACL shapes define what to index. Lucene stores one document per entity. SPARQL property functions expose search and facets.

1. Define shapes in config.ttl
## SHACL shape defines what to index
:GeochemShape
    sh:targetClass exg:GeochemistryAnalysis ;
    sh:property [
        idx:fieldName "element" ;
        idx:fieldType idx:KeywordField ;
        idx:facetable true ;
        sh:path ( sosa:observedProperty rdfs:label ) ;
    ] .
2. Lucene entity document (auto-built)
uri gswa:obs/surface_97080... stored
title "Bogada Bore Y53727" TextField
element "Au" KeywordField
procedure "XRF" KeywordField
lab "Genalysis" KeywordField
entityType "Geochemistry Analysis" KeywordField
3. Query via SPARQL property functions
PREFIX luc: <urn:jena:lucene:index#>

-- Search with filters
SELECT ?entity ?score ?totalHits
WHERE {
    (?entity ?score ?_lit ?totalHits)
        luc:query ('default' 'gold'
            '{"op":"=","args":[{"property":"element"},"Au"]}' 10)
}
-- Get facet counts
SELECT ?field ?value ?count
WHERE {
    (?field ?value ?count)
        luc:facet ('default' 'gold'
            '["element","procedure","lab"]' 10)
}

Full-Text Search with Filters

luc:query extends Jena text search with OGC CQL2-JSON filter arguments and a ?totalHits binding. Filters use CQL2-JSON for arbitrary boolean trees, range queries, and set membership. 

PREFIX luc:  <urn:jena:lucene:index#>

SELECT ?entity ?score ?totalHits
WHERE {
    (?entity ?score ?_lit ?totalHits)
        luc:query ('default' 'Maitland'
            '{"op":"=","args":[{"property":"element"},"Au"]}' 10)
}
ArgumentTypeDescription
queryStringStringLucene query (supports wildcards, phrases, field scoping)
filterCQL2-JSONOGC CQL2-JSON filter — {"op":"=","args":[{"property":"field"},"value"]}
limitIntegerMax result rows. ?totalHits still returns full count
Key Innovation
?totalHits is computed via IndexSearcher.count() — an O(1) operation that doesn't retrieve documents. This enables "Showing 10 of 1,234 results" without fetching all rows.
CQL2-JSON Filters
OGC CQL2-JSON — arbitrary boolean trees (and/or/not), comparisons, set membership (in), range queries (between), and pattern matching (like).

Compiled to Lucene queries via pushdown/residual split. Spatial ops stubbed for future implementation.

Faceted Navigation

luc:facet returns aggregate counts per field value. Powered by Lucene's SortedSetDocValuesFacetCounts for O(1) counting. 

PREFIX luc:  <urn:jena:lucene:index#>

SELECT ?field ?value ?count
WHERE {
    (?field ?value ?count)
        luc:facet ('default' '*'
            '["element","procedure","lab"]' 10)
}
ORDER BY ?field DESC(?count)
Performance
O(1) facet counting — pre-built DocValues structures, not document iteration. ~25% indexing overhead for DocValues creation at write time.
ArgumentDescription
indexIdIndex identifier (e.g. "default")
queryStringLucene query to scope facets
facetFieldsJSON array of field names
filterOptional CQL2-JSON filter (same as luc:query)
maxValuesMax values per field (0 = all)
minCountExclude rare values below threshold

Shared Execution

When luc:query and luc:facet appear in the same SPARQL query with matching parameters, they share a single Lucene execution — one index read, consistent snapshot. 

-- UNION pattern: N+M rows, not N*M
SELECT ?entity ?score ?totalHits
       ?field ?value ?count
WHERE {
  { (?entity ?score ?_lit ?totalHits)
        luc:query ('default' 'gold' 10) }
  UNION
  { (?field ?value ?count)
        luc:facet ('default' 'gold'
            '["element","procedure"]' 10) }
}
Execution Sequence
SPARQL
luc:query
SearchExec
Lucene
1. SPARQL engine evaluates luc:query
2. luc:query calls SearchExecution.getOrCreate(key)
3. SearchExecution runs Lucene query → hits + IndexReader
4. Returns hit URIs + scores to SPARQL
5. SPARQL engine evaluates luc:facet
6. luc:facet calls getOrCreate(same key) — reuses existing
7. Facet counts computed from cached IndexReader
8. Returns (field, value, count) to SPARQL

Sequence & Inverse Paths

SHACL sh:path expressions traverse multiple predicates or reverse direction. Indexed transparently — the SPARQL consumer never sees the traversal.

Sequence Path (drillhole example)
sh:path ( sosa:observedProperty rdfs:label )

Traverses Observation → ObservableProperty → Label at index time. The element field contains "Au", not a URI.
Multi-Hop Sequence Path
sh:path ( sosa:usedProcedure rdfs:label )

Traverses through to the procedure label. procedure field contains "XRF", "ICP", "MMI-M".
Inverse Path
sh:path [ sh:inversePath ex:authored ]

Traverses backwards. Indexes the author URI on the report entity.
Filter by Element (Sequence Path) 
PREFIX luc: <urn:jena:lucene:index#>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT ?entity ?score ?label
WHERE {
    (?entity ?score) luc:query
        ('default' '*' '{"op":"and","args":[{"op":"=","args":[{"property":"element"},"Au"]},{"op":"=","args":[{"property":"lab"},"ALS"]}]}' 5) .
    ?entity rdfs:label ?label .
}
ORDER BY DESC(?score)

Automatic Index Maintenance

ShaclTextDocProducer listens to every triple change. On add or delete of a relevant predicate or rdf:type, the entire entity document is rebuilt from the base dataset.

Design Choice
Full rebuild per entity change (not incremental field update) ensures correctness for multi-valued fields and sequence paths. For typical entities (<50 triples), rebuild is sub-millisecond.
Graph Support
Reads from union of default + named graphs. Entities indexed regardless of which graph the data is in.
Change Listener Flow
1 Triple add/delete arrives at DatasetGraphTextMonitor
2 super.add() updates base dataset first
3 ShaclTextDocProducer.change() fires
? Check: is predicate relevant (rdf:type or indexed field)?
4 Read all entity triples from union graph
5 Build Entity with all field values (PathEval for sequences)
6 Delete old Lucene doc, add new Lucene doc

Bounding-box filter via Lucene LatLonPoint. Scope search + facets to a geographic region.

  '{"op":"s_within","args":[{"property":"location"},{"type":"Polygon","coordinates":[[...]]}]}')
DrillSideways
Single-pass combined search + facet collection. When filtering by element=Au, the element facet still shows counts for all elements.
